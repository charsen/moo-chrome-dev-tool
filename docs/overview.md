# Moo Dev Tool — 研发立项说明

> 本文用于企业内部研发立项评审：一句话介绍、立项背景、分模块说明、全功能点清单、质量与安全设计、现状与规划。
> 技术细节索引：用户手册见 [README.md](../README.md)，后端接入协议见 [SERVER_INTEGRATION.md](SERVER_INTEGRATION.md)，禅道接入见 [ZENTAO_SETUP.md](ZENTAO_SETUP.md)，版本史见 [CHANGELOG.md](../CHANGELOG.md)。

---

## 一、一句话介绍

**Moo Dev Tool 是一个团队内部使用的 Chrome 浏览器扩展：测试/开发/设计在出问题的页面上「一键截图标注（或录屏）→ 填一行标题 → 提交」，扩展自动把现场证据（截图、录屏、网络请求、JS 报错、页面元素、环境信息）打包上报到团队的 bug 系统（自建 webhook 后端或禅道），开发不用再追问「你怎么操作的、请求长什么样」。**

---

## 二、立项背景：要解决什么问题

**报 bug 的隐性成本一直很高。** 传统流程下，测试发现「按钮点了没反应」要做五件事：截图、开 DevTools 抄请求、复制错误堆栈、回忆操作路径、写一段开发能看懂的描述——单条 bug 报障 5-15 分钟，且关键证据（当时的请求体/响应、控制台报错）经常缺失，开发被迫往返追问，一条 bug 的沟通成本轻松翻倍。

**现有工具不贴合内部研发场景：**

| 替代方案 | 差距 |
|---|---|
| 手工截图 + IM 描述 | 无请求/报错现场，全靠追问 |
| 商业 SaaS 报障工具 | 数据出公司、按席位收费、对接不了内部禅道 |
| Sentry 类被动监控 | 抓异常不抓「人的操作意图」，测试无法主动圈问题 |
| 浏览器自带录屏/截图 | 不带网络与报错上下文，仍要手工归档到工单 |

**本项目的定位**：把「人发现的问题」和「机器记录的现场」在浏览器里合流，一次提交直达团队 bug 看板。数据走团队自己的后端（自建 webhook / 禅道 / moo-scaffold 生态），不出企业边界。

## 三、方案概览

**产品形态**：Chrome MV3 扩展（Manifest V3，Chrome 111+），Vite 5 + Vue 3 + TypeScript 实现，约 2 万行源码 / 97 个源文件。

**工作原理（数据流）**：

```
宿主页面 (MAIN world hook)            内容脚本 (ISOLATED world)
  fetch/XHR 抓包 ─┐                     悬浮球 / 截图标注 / 提交弹窗 / 录制条
  JS 报错捕获    ─┼─ postMessage ──▶    环形缓冲 + 脱敏
  SPA 路由信号   ─┘                          │ chrome.runtime 消息
                                             ▼
                                  Service Worker (后台中枢)
                                    项目匹配 / 动态注册 / 提交路由
                                    重试队列 / 状态回查 / 版本检查 / badge
                                       │                    │
                              IssueAdapter 抽象层      offscreen 文档
                              ┌────────┴────────┐      (录屏编码器)
                              ▼                 ▼
                        通用 Webhook        禅道 API
                     (自建后端/云端)     (v2/v1 双轨直连)
```

**三个使用入口**：① 页面右下角悬浮球 + 快捷键（报障主入口）；② F12 DevTools「Moo」面板（4 Tab：概览/历史/环境/设置）；③ 浏览器工具栏 popup + 独立「工作区」浮窗（不开 DevTools 也能完成全部配置与回看）。

## 四、分模块介绍

按源码目录划分，每个模块职责单一、消息协议强类型（`types/messages.ts` 统一 dispatch）。

| 模块 | 职责 | 要点 |
|---|---|---|
| **`src/injected/`**（MAIN world） | 注入宿主页面真实 JS 世界的钩子 | monkey-patch `fetch` / `XMLHttpRequest` 抓请求与响应（20KB 截断）、`window.error` + `unhandledrejection` 抓报错（100ms 同消息去重）、hook `history.pushState/replaceState` 感知 SPA 路由。自包含 IIFE、零 chrome API 依赖；幂等守卫防重复注入导致双采集 |
| **`src/content/`**（ISOLATED world） | 页面上的全部交互 UI | 悬浮球（拖拽/多项目切换/当前站隐藏）、截图标注器（画框/文字/撤销/复制/下载）、元素选取器、提交弹窗（webhook/禅道两套表单）、录制条。全部 UI 挂在 **closed shadow DOM** 内，宿主页脚本读不到扩展内容；接收 MAIN world 消息做 source/origin/shape 三重校验 |
| **`src/background/`**（Service Worker） | 无界面后台中枢 | onMessage 路由（sender 严格校验）→ 各 handler；内容脚本**动态注册**（按用户配置的 URL 才注入，非全站）；提交编排（写历史/刷 badge/失败入队）；重试队列（写锁 + 冷却 + 成功回填）；定时任务（5min 重试 / 24h 版本检查）；录屏控制（用户手势链、双重防护） |
| **`src/background/zentao/`** | 禅道 API 客户端 | v2 token 鉴权 + **v1 fallback 双轨**（多实例 schema 方差是实测痛点，硬规则禁止 v2 单轨）、401 自动重登、403 cookie 级联兜底、产品/模块/人员自动发现（24h 缓存）、multipart 建单与附件上传 |
| **`src/adapters/`** | 上报通道抽象（IssueAdapter） | `submit / fetchStatus / serializeForRetry / retryFromPayload` 四个契约把「通道差异」收口：现有 webhook、禅道两个实现；未来接 GitHub/Jira 等只需新增 adapter，编排层零改动 |
| **`src/offscreen/`** | 录屏编码器（MV3 要求独立文档） | tabCapture 流 → MediaRecorder 编码 → base64 回传（46MB IPC 上限保护）；显式状态机（idle/starting/recording/stopping）+ 35s 自停三层保险；keepalive 防 Chrome 回收 |
| **`src/devtools/`** | F12「Moo」面板 | 4 Tab：**概览**（实时请求/报错时间线、详情、JSON 高亮搜索）、**历史**（30 条提交记录、远端状态回查、一键重提）、**环境**（项目/服务器 CRUD、URL 匹配规则、禅道凭证、payload 模板）、**设置**（脱敏 keys、抓取开关、重试队列管理、配置导入导出） |
| **`src/popup/` + `src/options/`** | 工具栏弹窗 + 独立工作区 | popup：当前页匹配状态、录屏开关、悬浮球当前站隐藏、权限一键启用、版本检查；工作区：独立浮窗承载与 DevTools 完全相同的 4 Tab（给不习惯开 F12 的测试/设计用） |
| **`src/storage/`** | chrome.storage 封装 | 配置与历史的读写、**read 边界统一 normalize**（老数据/异常数据兜底）、写互斥锁（防多窗口 last-write-wins）、配额超限逐条降级 |
| **`src/composables/` + `src/utils/`** | 共享逻辑 | 自动保存防抖、版本检查、token/敏感字段脱敏、模板渲染、curl 生成、缩略图压缩、focus 管理（宿主页焦点战争对策）等 30+ 个单测覆盖的纯函数模块 |
| **`src/i18n/`** | 文案集中管理 | 全部用户可见文案走 `t()`，为多语言留口子 |

## 五、功能点全集与说明

### 5.1 现场采集（自动，零操作）

| 功能 | 说明 |
|---|---|
| 网络请求抓取 | 抓当前页最近 N 条 fetch/XHR（默认 50，可配），含 method/URL/请求头/请求体/响应头/响应体/耗时/状态码；相对路径自动补全为完整 URL |
| 响应体截断保护 | 单条 body 20KB 截断，防超大响应拖垮页面 |
| JS 报错抓取 | 未捕获异常 + Promise rejection，含堆栈、文件、行列号；高频同错 100ms 去重防刷屏 |
| SPA 路由感知 | 前端路由切换（pushState/replaceState/popstate/hashchange）自动重新匹配项目，单页应用内导航无感 |
| 存储快照 | 按项目配置的**白名单 key** 抓 localStorage/sessionStorage（如当前用户 id），非白名单一概不读 |
| 环境信息 | URL、UserAgent（解析成可读浏览器/系统名）、视口尺寸、时间戳自动附带 |

### 5.2 报障流程（人工，三步完成）

| 功能 | 说明 |
|---|---|
| 悬浮球 | 配置命中的页面右下角出现，可拖拽、可对当前站点临时隐藏（会话级）；多项目命中同一 URL 时弹选择器 |
| 页面快捷键 | 截图报障支持键盘直达；`⌥⇧R / Alt+Shift+R` 启动录屏（Chrome 要求录屏必须键盘手势）；`⌥⇧M` 打开 popup |
| 截图 | captureVisibleTab 全页截图；**密码输入框自动遮挡**（默认开，可按项目关闭） |
| 标注器 | 截图上画矩形框、写文字、撤销；标注后的图可**直接复制到剪贴板**（HTTPS 页）或**下载 PNG**（任意页），不依赖 chrome.downloads 权限 |
| 录屏 | 最长 30 秒自动停（content 30s + offscreen 35s + SW alarm 三层保险）；录制中鼠标点击显示红圈涟漪（开发能看清点了哪）；rec 条实时计时、可手动停止/取消 |
| 元素选取 | 「选元素」进入拾取模式，点中的元素自动生成 selector 附进报告 |
| 提交弹窗 | 标题/描述、请求与报错逐条勾选、截图缩略图、录屏附带开关、目标服务器下拉；禅道项目额外提供 类型/严重/优先级/指派人/模块 下拉（数据从禅道实例自动拉取）；提交结果展示远端单号 + 「禅道里看」直达链接 |
| Payload 预览 | webhook 提交前可预览渲染后的最终 JSON，所见即所发 |
| 宿主页焦点对策 | 页面 modal/富文本抢焦点场景下，扩展弹窗仍可正常输入（focus 偷取 + 守护监听，实测三轮迭代收敛） |

### 5.3 上报通道

| 功能 | 说明 |
|---|---|
| 通用 Webhook | POST 到团队自建后端；**payload 模板可视化自定义**（`{{title}}` `{{image}}` `{{token}}` 等占位符，JSON 上下文自动转义防结构破坏）；支持 JSON 与 multipart 两种体；自定义请求头（key 合法性校验、防撞键） |
| 禅道直连 | 免后端开发，直接在禅道建 bug 单：v2 API 优先 + **v1 自动 fallback**（兼容多版本/多实例 schema 方差）、401 自动重新登录、v1 403 自动带浏览器 cookie 级联重试、附件（截图/录屏）multipart 上传、产品/模块/人员自动发现并缓存 |
| 生态配套 | 后端可用配套 Laravel 包 moo-scaffold（自带接收接口 + 管理后台），或 moo-scaffold-cloud 云端 `todos/intake` 直接对接；协议公开（两个 HTTP 接口），任何语言可自行实现 |
| Token 安全 | token 走 POST body 注入（不进 URL，避免 access log / referer 残留）；状态回查同样 POST 带 body token |

### 5.4 提交后闭环

| 功能 | 说明 |
|---|---|
| 本地历史 | 最近 30 条提交记录（截图自动压缩成缩略图省配额），含完整提交内容 |
| 远端状态回查 | 历史列表一键同步远端处理状态（webhook 走 `status-public` 协议、禅道走 getBug），显示「处理中/已解决/已关闭/已删除」 |
| 一键重提 | 历史记录直接重新提交（含当初选的禅道 类型/严重/优先级/指派人/模块 快照，不回落默认值） |
| 失败重试队列 | 网络错/5xx 自动入队（上限 50 条、单条 1MB），5 分钟周期 + SW 唤醒自动重试（最多 5 次）；**90s 全局冷却 + 60s 单条冷却**双层防重复建单；**重试成功自动回填历史状态**；设置页可看队列明细、手动立即重试、删单条/清空 |
| 失败角标 | 工具栏图标 badge 显示 24h 内失败计数（红），有新版本时显示升级提示（蓝），让失败不被无视 |

### 5.5 配置与管理

| 功能 | 说明 |
|---|---|
| 多项目 | 一套扩展管理公司内任意多个系统：每个项目绑定 URL 匹配规则（`https://*.example.com/*` 通配），悬浮球按当前网址自动命中 |
| 多服务器 | 每个项目可配多个上报服务器（开发/预发/线上），提交时下拉切换，default 可指定 |
| 项目级开关 | 请求抓取、报错抓取、缓冲区大小、密码遮挡、存储白名单逐项目独立配置 |
| 脱敏规则 | 请求头/请求体敏感字段自动打码（默认覆盖 authorization/cookie/token/secret 等 12 类，可增删）；URL query 同样处理；用户显式清空可保持（不被默认值覆盖） |
| 配置导入导出 | 一键导出 JSON 给同事（**密码等敏感字段自动剥离**），导入有结构校验与确认 |
| 自动保存 | 环境配置 800ms 防抖自动落盘 + 关闭前强制 flush，不丢编辑 |
| URL 智能建议 | 新建项目自动带入当前页 URL 生成匹配规则；当前页不命中任何项目时顶部提示一键追加 |

### 5.6 权限与升级

| 功能 | 说明 |
|---|---|
| 最小权限设计 | `host_permissions` 为 **optional**（用户主动授权才能向页面注入/上报）；内容脚本**动态注册**（只注入配置命中的站点，不做全站注入）；录屏 tabCapture 同为可选权限，popup 一键启用 |
| 版本检查 | SW 每 24h 检查发布仓库最新 release（可手动触发），新版在 popup/工作区显示下载链接；检查失败明确显示「检查失败」而非谎报最新（三态语义） |
| 一键升级 | 下载解压覆盖后点「重新加载」即完成（等价 chrome://extensions ↻）；升级成功弹「✓ 已升级到 vX.Y.Z」确认 toast；录屏进行中阻止误升级 |

### 5.7 安全与隐私设计（贯穿性）

| 设计 | 说明 |
|---|---|
| 数据不出企业 | 所有上报数据只发往用户自己配置的后端地址，扩展不内置任何第三方数据收集 |
| closed shadow DOM | 扩展 UI（含用户输入的标题/描述、截图）对宿主页脚本不可见 |
| 消息三重校验 | MAIN→ISOLATED：source + origin + payload shape；扩展内部消息：sender.id 严格等于自身扩展 id，防第三方扩展窃取/伪造 |
| 主动脱敏 | 请求体/头/query 的敏感字段在**进入本地缓冲前**就打码（非提交时才处理），非 JSON 体同样覆盖 |
| 注入幂等 | 重复注入有 window flag 守卫，不会双 hook 双采集 |
| 发版红线 | 发布流水线内置 PII 扫描（黑名单 + 手机号/邮箱/内网 IP/证件号模式扫描），命中即中止 |

## 六、技术栈与运行要求

- **运行环境**：Chrome / Edge 等 Chromium 系 111+（Manifest V3）
- **技术栈**：Vite 5 · Vue 3（Composition API）· TypeScript（strict + noUncheckedIndexedAccess）· @crxjs/vite-plugin · pnpm
- **测试**：Vitest 单元测试 + Playwright 端到端（真实扩展加载）双断面；chrome-devtools MCP 真机验证作发版前补充断面
- **零运行时三方依赖面**：UI 仅 Vue；后台/注入层为零依赖纯 TS（供应链风险面小）

## 十、立项结论建议

本项目以极小的运行时依赖与企业内自有数据闭环，解决「报 bug 现场还原成本高」这一高频痛点；架构上通道抽象（IssueAdapter）保证了对接任意内部工单系统的扩展性，质量体系（849 个自动化用例 + 红→绿回归制度 + PII 发版红线）已达到可持续维护标准。建议立项并按「短期清账 → CWS 上架 → 按需扩通道」节奏推进。
